## ğŸ› ï¸ Development Workflow

### Getting Started
1. Fork the repository
2. Clone your fork: `git clone https://github.com/yourusername/al-nur-al-khafi.git`
3. Create a branch: `git checkout -b feature/your-feature-name`
4. Make your changes
5. Commit following conventional commits
6. Push to your fork
7. Open a pull request

### Commit Message Format
We follow [Conventional Commits](https://www.conventionalcommits.org/) specification:

```
type(scope): description

body (optional)

footer (optional)
```

Types:
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation changes
- `style`: Code formatting, missing semicolons, etc.
- `refactor`: Code refactoring
- `test`: Adding or modifying tests
- `chore`: Build process or auxiliary tool changes
- `perf`: Performance improvements
- `ci`: CI related changes
- `sacred`: Religious content updates (special type)

Example:
```
feat(biography): add detailed account of Hijrah journey

Added comprehensive timeline events for the Migration
Including key dates, companions involved, and challenges faced

Closes #123
```

### Pull Request Process
1. Ensure your code follows our style guides
2. Run all tests and ensure they pass
3. Update documentation if needed
4. Include screenshots for UI changes
5. Link related issues in PR description
6. Request review from maintainers

### Code Review Guidelines
Reviewers should:
- Focus on code quality and maintainability
- Check for security vulnerabilities
- Ensure religious content accuracy
- Verify performance implications
- Maintain respectful communication
